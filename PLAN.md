# creditwhere-skill 项目计划书

> 一个 AI Agent Skill。Agent 加载后获得「评估 Git 仓库贡献质量」的能力。
> 不做规则引擎，不写分类器——Agent 自己跑 git 命令搜集证据，用自己的模型做全部判断。

---

## 一、项目概述

### 做什么

开发一个 Skill 文件（SKILL.md），安装到任意 AI Agent（Hermes、Codex 等）后，Agent 即可对 Git 仓库执行完整的贡献质量分析。

用户说「帮我分析下这个仓库谁贡献最多」，Agent 就会：

1. 跑 git 命令收集提交历史、diff、blame 数据
2. 用自己的推理能力判断每个作者的变更实质性、代码存活率、提交模式
3. 做交叉对比，发现改写冲突、低实质提交等异常
4. 输出一份结构化的证据附注报告

### 不做

- 不写任何 Python 分类逻辑或正则规则
- 不写评分公式代码
- 不做 CLI 工具、不做 pip 包
- 不依赖外部 API（Agent 自己的模型就是分析引擎）

### 交付物

```
creditwhere-skill/
├── SKILL.md                      # 主文件：流程指南 + 报告模板元指令
├── prompts/
│   ├── file-importance.md        # 文件重要性评估提示词
│   ├── author-deep-dive.md       # 单作者深度分析提示词
│   └── cross-comparison.md       # 交叉对比提示词
├── examples/
│   └── sample-report.md           # 完整示例报告
└── PLAN.md                       # 本文件
```

## 二、核心设计

### 2.1 架构原则

```
用户一句话 ──▶ Agent 加载 SKILL.md ──▶ Agent 自主执行 6 步流程 ──▶ 输出报告
                         │
                         ├── 用 terminal 工具跑 git 命令
                         ├── 用自己的模型做语义判断
                         └── 按模板格式化输出
```

**所有判断发生在 Agent 的推理过程中。** Skill 文件不包含任何硬编码逻辑，只包含：
- 告诉 Agent 每一步做什么、用什么工具
- 给 Agent 的提示词模板（判断准则、证据格式要求）
- 输出报告的格式规范

### 2.2 为什么这个设计可行

现代 Agent 模型已经具备以下原生能力：
- 阅读和理解 git diff 内容，判断一个变更是「修 bug」还是「改格式」
- 对比两个 commit 之间的代码，判断是否属于「重写」
- 分析 commit message 语义，判断是否敷衍
- 理解项目目录结构，判断哪些是核心模块
- 在多个数据点之间做交叉推理

Skill 文件的工作不是教模型「怎么判断」，而是告诉模型「按什么顺序、关注哪些维度、输出什么格式、引用什么证据」。

## 三、SKILL.md 详细设计

### 3.1 结构

```markdown
---
name: creditwhere
description: >
  Evaluate contributor effort quality in a Git repository.
  Uses git commands to gather evidence, then applies LLM reasoning.
  Outputs a structured report with evidence annotations.
triggers:
  - "评估贡献"
  - "creditwhere"
  - "谁贡献最多"
  - "分析仓库贡献"
---

# creditwhere

## What this skill does
（简短说明）

## Workflow
### Step 1: Confirm scope
### Step 2: Collect git data
### Step 3: Assess file importance
### Step 4: Deep-dive per author
### Step 5: Cross-comparison
### Step 6: Generate report

## Report Template
（完整的报告 Markdown 模板）

## Principles
（评估准则）
```

### 3.2 六步流程说明

**Step 1：确认范围**

Agent 先确认分析参数。如果用户没有明确给出，主动询问：

- 仓库路径
- 时间范围（默认全部历史）
- 排除路径（默认排除 node_modules、vendor、dist、*.lock）
- 关注作者（默认全部）

**Step 2：采集数据**

分三轮执行，控制每次返回的数据量：

| 轮次 | 命令 | 目的 |
|------|------|------|
| 第一轮 | `git shortlog -sne --all` | 获取全部作者和提交数 |
| 第一轮 | `git log --oneline --all \| wc -l` | 总提交数 |
| 第二轮 | 对每人依次：`git log --author="NAME" --format="%H %s" --all` | 某人的全部提交摘要 |
| 第二轮 | 对每人依次：`git log --author="NAME" --numstat --all` | 某人的文件级变更统计 |
| 第三轮 | `git blame <core_file>` | 核心文件的当前行归属 |

关键约束：**每次只处理一个作者的数据，分析完再进下一个。** 不要让所有作者的数据同时堆在上下文里。

**Step 3：评估文件重要性**

Agent 扫描仓库的文件和目录结构后，用自己的理解给文件分级：

```
指令：基于目录结构判断各类文件的重要性。
输出：表格，每行包含路径模式、重要性标签、理由。

示例：
src/core/engine.py → 核心 — 主要业务逻辑入口
src/utils/helpers.py → 高 — 被多处引用的工具函数
tests/ → 高 — 测试代码
README.md → 中 — 文档
package-lock.json → 忽略 — 自动生成物
```

注意：不要让 Agent 跑 `ls` 扫全部文件，用 `git log --numstat` 的输出来识别热点文件即可。

**Step 4：单作者深度分析（核心）**

这是最关键的步骤。Agent 加载 `prompts/author-deep-dive.md` 中的提示词模板，对每个作者依次回答五个问题：

| 维度 | 问题 | 要求 |
|------|------|------|
| 实质度 | 这个人的变更中，逻辑变更 vs 格式/注释变更的比例？ | 举 3-5 个代表性提交，各标注实质度：高/中/低 |
| 核心参与 | 是否触及核心模块？还是只在边缘文件？ | 列出涉及的核心文件路径 |
| 存活率 | TA 引入的代码有多少存活到 HEAD？ | 比较 `git blame` 结果与引入记录 |
| 提交模式 | message 质量、颗粒度、时间分布是否正常？ | 具体例子 |
| 整体评价 | 用 2-3 句话总结 | 必须引用上述证据 |

**证据标注格式（强制）：**
```
每次给出判断时，必须用以下格式附注证据：
- 提交示例：`commit abc123 ("fix bug") — 实质度：低 — 只改了 2 行注释`
- 文件归属：`src/core/handler.py:120-145 — Alice`
- 存活率：`Alice 引入的 340 行中，287 行存活到 HEAD（84%）`
```

**Step 5：交叉对比**

所有作者分析完后，Agent 加载 `prompts/cross-comparison.md`，做横向比较：

1. 代码重叠：是否有人反复改同一批文件？
2. 改写检测：有人覆盖了别人的代码吗？（对比 blame 变化）
3. 异常识别：提交多但实质度低、存活率异常低、message 全是两个字
4. 贡献排序：综合实质度 + 核心模块 + 存活率，给出排序及理由

**Step 6：生成报告**

Agent 按照 SKILL.md 中的报告模板，将前面所有分析结果填入格式化输出。报告包含六个部分：

1. 仓库概览
2. 文件重要性地图
3. 每个作者的详细分析（五维度 + 证据）
4. 交叉对比总表
5. 异常标记汇总
6. 局限性声明

## 四、提示词模板设计

### 4.1 prompts/author-deep-dive.md

```
你正在评估一个开发者在 Git 仓库中的贡献质量。
以下是该仓库中此人的全部提交摘要和关键 diff。
请用你的理解能力回答以下五个问题。

## 数据
[此处插入 git log 输出和关键 diff]

## 评估要求

### 1. 变更实质度
仔细阅读每次提交的 diff 内容。区分：
- 逻辑变更：修 bug、加功能、改算法、改行为
- 重构：改结构但行为不变（重命名、提取函数、移动代码）
- 格式/注释：只改空格、换行、注释、文档字符串、lint
- 配置/依赖：改配置文件、锁文件、CI

列出 3-5 个最有代表性的提交，每个标注实质度（高/中/低）和理由。
最后给出整体估计：大约 X% 的变更是实质性变更。

### 2. 核心模块参与
根据之前分析的文件重要性地图，判断此人的改动集中在哪些模块。
是核心逻辑、测试、还是边缘文件？

### 3. 代码存活率
用 git blame 结果对比此人的引入记录。
计算：存活行数 / 引入行数。标注哪些代码被保留了，哪些被重写了。

### 4. 提交模式
- 提交信息：有意义的描述 vs 敷衍的"fix""update"
- 颗粒度：每次提交改 50+ 行核心逻辑 vs 每次改 1-2 行格式
- 时间分布：均匀 vs 集中在某些时段

### 5. 整体评价（2-3 句）
基于以上四个维度的证据，用自然语言概括此人的贡献特征。

## 格式要求
- 每个判断必须附注证据（commit hash 或文件路径）
- 用「可能」「大概率」而非「一定」当证据不足时
- 不要给出没有证据支持的结论
```

### 4.2 prompts/cross-comparison.md

```
你已经逐个分析了仓库中所有作者的贡献。现在对全部数据做横向比较。

## 任务

### 1. 代码重叠分析
找出被多个作者频繁修改的文件，判断是协作还是冲突。

### 2. 改写检测
基于 git blame 变化，判断是否有人重写了别人的代码。
列出具体案例：谁改了什么、谁的代码被覆盖了、这发生在哪个 commit。

### 3. 异常标记
从以下信号中识别异常：
- 提交数多但实质度低（例如 50 个提交中 40 个只改格式）
- 代码存活率明显低于同仓库平均水平
- commit message 异常简短或统一（全部是 "fix" / "update"）
- 只改测试文件或只改文档

对每个异常标记，必须提供具体证据。

### 4. 贡献排序
综合实质度、核心模块参与度和存活率，给出作者贡献排序。
必须用 1-2 句话解释排序理由，不能用单一数字代替。

## 格式要求
输出为 Markdown 表格：

| 作者 | 实质变更 | 核心模块 | 存活率 | 异常标记 | 贡献排序 | 排序理由 |
|------|----------|----------|--------|----------|----------|----------|
| ... | ... | ... | ... | ... | ... | ... |
```

### 4.3 prompts/file-importance.md

```
你正在分析一个 Git 仓库。以下是该仓库中实际被修改过的文件列表。
请根据你的理解，判断各类文件的重要性。

## 判断准则

- 核心源码：实现主要业务逻辑或算法
- 重要工具：被多处引用的公共模块
- 测试：测试代码
- 配置：构建、部署、依赖声明
- 文档：README、注释、API 文档
- 生成物：锁文件、构建产物、vendor 目录

## 输出格式

| 路径模式 | 重要性 | 理由 |
|----------|--------|------|
| src/core/* | 核心 | 主要业务逻辑 |
| src/utils/* | 高 | 公共工具模块 |
| tests/* | 高 | 测试代码 |
| docs/* | 中 | 文档 |
| package-lock.json | 忽略 | 自动生成物 |
```

## 七、注意事项

### 给开发者的提醒

1. **提示词是核心产品**
   这个项目的「代码」就是提示词。每改一次提示词，相当于改了核心逻辑。提示词要反复测试、对比效果。

2. **控制上下文长度**
   大仓库的 git log 输出可能非常长。SKILL.md 里的流程设计必须强制 Agent 分批处理——每次只分析一个作者、每次不加载全量 diff。如果不加这个约束，中型仓库就会超出上下文窗口。

3. **证据格式是铁律**
   提示词里反复强调「每个判断必须附注 commit hash」不是啰嗦——Agent 模型偶尔会「感觉 Bob 贡献少」但说不出来具体原因。强制格式化证据可以大幅减少这种空话。

4. **模型差异**
   同样的提示词，在 Claude 和 GPT 和 DeepSeek 上的表现可能不同。如果发现某个模型输出质量差，检查是不是提示词中有些「隐含假设」只对特定模型有效。

5. **不要加规则**
   你可能忍不住想加几条正则规则来提高准确率。克制住。这个项目的价值就在于「不用规则也能做」。如果加了规则，后面就会一直加，最后变成另一个 CLI 工具。

### 给 Agent 的注意事项

这些内容写在 SKILL.md 中，是给 Agent 看的：

- 所有判断必须有证据，没有证据的结论不出现在报告中
- 不确定时用「可能」「大概率」，不要用「一定」
- 每次只分析一个作者，分析完再处理下一个
- git blame 结果中如果出现多个作者改同一行，标注「共同归属」
- 报告结尾必须包含局限性声明，不能省略
- 不要输出「摸鱼」「划水」等价值判断词汇；用「低实质提交比例偏高」等描述性语言

## 八、验收标准

用一个 3+ 作者、100+ commit 的真实 Git 仓库测试。Agent 加载 SKILL.md 后，用户说「帮我分析贡献」，Agent 应该：

- [ ] 自主完成六个步骤，不卡在某个步骤等用户提示
- [ ] 报告中每个判断附有至少一条具体证据
- [ ] 能正确区分「核心贡献者」和「边缘贡献者」
- [ ] 能标记出明显的低实质提交
- [ ] 交叉对比部分有实质性发现，而不是「无明显异常」敷衍
- [ ] 局限性声明完整
- [ ] 报告整体读起来像一份人写的分析，而不是一个模板填空
