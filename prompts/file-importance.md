# 文件重要性评估提示词

你正在分析一个 Git 仓库。下面给你的不是完整文件树，而是从 `git log --numstat` 中抽取出来的实际改动热点文件。

你的任务不是打分，而是做语义分级：哪些路径更可能代表核心业务逻辑，哪些更像测试、配置、文档或生成物。

## 输入

- 改动频繁的文件路径列表
- 可选：仓库名称、少量目录上下文、主要语言

## 判断准则

- `核心`：主要业务逻辑、算法、主流程、关键服务、核心数据结构
- `高`：公共工具模块、集成层、重要测试、关键脚本
- `中`：普通配置、文档、次要工具、样例代码
- `低`：边缘脚本、一次性迁移、演示文件
- `忽略`：锁文件、构建产物、vendor、自动生成代码

优先考虑这些信号：

- 路径语义：如 `src/core/`、`backend/service/`、`pkg/engine/`
- 文件名语义：如 `router`、`engine`、`scheduler`、`handler`
- 测试位置：`tests/`、`spec/`、`__tests__/`
- 生成物特征：`dist/`、`vendor/`、`*.lock`、`coverage/`

不要做的事：

- 不要发明数字权重或总分
- 不要因为文件改动次数高就自动判为核心
- 不要在缺乏证据时把所有源码都判成核心

## 输出格式

输出一个 Markdown 表格，每行尽量归纳成路径模式，而不是重复单个文件：

| 路径模式 | 重要性 | 理由 |
|----------|--------|------|
| `src/core/*` | 核心 | 主业务流程与核心逻辑集中在这里 |
| `src/utils/*` | 高 | 被多处复用的工具模块 |
| `tests/integration/*` | 高 | 关键行为验证 |
| `docs/*` | 中 | 文档，对功能实现不是直接贡献 |
| `package-lock.json` | 忽略 | 自动生成依赖锁文件 |

## 额外要求

- 结论要可复用到后续作者分析中。
- 如果某一类路径的重要性不确定，明确写出不确定性来源。
- 尽量控制在 5-12 行，突出真正重要的路径模式。
