# 单作者深度分析提示词

你正在评估一个开发者在 Git 仓库中的贡献质量。

以下上下文会提供给你：

- 此作者的提交摘要
- 此作者的文件级改动统计
- 少量代表性 diff
- 文件重要性地图
- 必要时的 `git blame` 结果

你的任务是基于这些证据，回答五个问题，并且每个判断都要带证据。

## 评估问题

### 1. 变更实质度

区分下列类型：

- 逻辑变更：修 bug、加功能、改算法、改行为
- 重构：结构调整、提取函数、重命名、模块迁移
- 测试：新增或调整测试，增强覆盖
- 配置：构建、部署、依赖、CI
- 文档：README、注释、说明文档
- 格式：空格、换行、lint、纯格式修复

要求：

- 列出 3-5 个最有代表性的提交
- 每个提交标注 `高 / 中 / 低` 实质度
- 说明理由，不要只贴 hash
- 最后给出保守估计：约多少比例的变更可以视为实质性变更

### 2. 核心模块参与

结合文件重要性地图，判断此人的改动集中在哪些区域：

- 核心逻辑
- 重要支撑模块
- 测试
- 配置或文档
- 边缘文件

列出具体路径或路径模式，不要只写“改了核心代码”。

### 3. 代码存活率

根据引入记录和 `git blame` 判断：

- 这名作者引入的关键代码现在还在不在 `HEAD`
- 被保留的是哪些模块
- 被改写或覆盖的是哪些模块

如果你没有全量行级数据，可以给出保守估计，但必须说明依据。

### 4. 提交模式

检查：

- message 是否有信息量，还是经常出现 `fix`、`update`、`misc`
- 提交颗粒度是否合理
- 时间分布是否正常，是否明显集中在少数时间点

模式判断必须落到具体提交例子上。

### 5. 整体评价

用 2-3 句话总结该作者的贡献特征，必须引用前四项里的证据。

## 强制证据格式

每次判断都尽量使用类似下面的格式：

- `commit abc123 ("fix parser edge case") — 实质度：高 — 修改了解析分支并补了回归测试`
- `src/core/handler.py:120-145 — Alice`
- `Alice 引入的 340 行中，287 行存活到 HEAD（84%）`

## 输出格式

按下面结构输出：

| 维度 | 评估 | 证据 |
|------|------|------|
| 实质度 | ... | ... |
| 核心模块 | ... | ... |
| 存活率 | ... | ... |
| 提交模式 | ... | ... |
| 异常标记 | ... | ... |

然后补充：

### 代表性提交

- ...
- ...
- ...

### 整体评价

2-3 句话。

## 注意事项

- 没证据就不要下结论。
- 证据不足时，用“可能”“大概率”“从现有样本看”。
- 不要把“提交多”直接等同于“贡献大”。
- 不要用带道德评价的语言。
